# 需求和设计文档

## 1. 文档目的

本文件用于当前项目阶段的统一交接，替代此前分散的多份 2026 设计稿。目标是让后续接手者只看这一份文档，就能快速理解：

1. 赛题到底要求什么
2. 项目已经做到什么阶段
3. 系统应该怎么继续做
4. 哪些页面和能力是主线，哪些只是辅助页
5. 后续迭代应该优先补什么，不该再往哪里乱堆

## 2. 赛题理解

根据 `赛题.pdf` 和 `赛题相关数据下载路径文档.pdf`，本项目对应的是 2026 年中国大学生计算机设计大赛大数据主题赛题：

`智能体赋能的企业运营分析与决策支持系统`

赛题的核心不是做一个泛聊天机器人，而是围绕某个特定行业或领域，利用公开、合法、可追溯的数据，构建一个可交互、可分析、可给出决策支持结果的系统。

赛题给出的重点方向包括：

1. 智能体主导的智能问数服务
2. 企业运营评估与问题诊断
3. 风险与机会洞察
4. 定制化内容生成与来源追溯
5. 智能体自身的技术优化与效率提升

赛题允许并鼓励补充其他合法合规数据，但必须遵守数据安全和信息披露要求，不能使用来源不明或违规数据。

## 3. 推荐项目定位

本项目不应再定义为“多 agent 演示平台”或“企业分析聊天壳子”，而应定义为：

`面向企业运营分析、风险判断和管理决策的企业决策智能系统`

这个定位同时覆盖三类价值：

1. 对用户：能问、能看、能追证据、能得到建议
2. 对接手者：能快速看懂产品结构和后端主链路
3. 对工程：能体现传统应用、Agent、大模型、深度学习和大数据工程的协同

## 4. 角色与使用场景

### 4.1 企业管理者

核心诉求：

1. 快速理解单个企业的经营状态
2. 发现风险点和改进方向
3. 查看可执行的管理动作

### 4.2 投资研究者

核心诉求：

1. 快速提问
2. 对比企业
3. 看结论对应的证据来源

### 4.3 接手者 / 管理员

核心诉求：

1. 理解系统整体架构
2. 理解数据链路和工程能力
3. 理解当前主线、缺口和继续实现的优先级

## 5. 总体需求

### 5.1 业务目标

系统应当围绕企业运营分析形成闭环：

1. 汇聚多源公开数据
2. 支持自然语言提问
3. 自动识别企业、任务意图和分析路径
4. 产出带证据的分析结果
5. 支持风险判断、同业比较和内容导出
6. 对结果进行基本校验和审计

### 5.2 产品目标

产品面必须控制复杂度，不能再做成长文档式页面。主产品线应收敛为：

1. 企业问答系统
2. 数字人页面
3. 企业分析工作台
4. 企业对比判断
5. 业务总览

其中：

- `企业问答系统` 是主入口
- `数字人页面` 当前只保留独立入口和预留位
- `企业分析工作台` 负责深度报告
- `企业对比判断` 负责横向比较
- `业务总览` 面向用户展示当前能力、数据可信度和结果可用性

### 5.3 技术目标

系统要体现四层能力：

1. 传统应用：登录、线程、页面、导出、状态管理、鉴权
2. Agent：意图识别、技能路由、执行编排、过程回显
3. 大模型 / 深度学习：理解、生成、检索重排、分类、风险评分
4. 大数据工程：采集、清洗、切片、索引、计算、治理、审计

## 6. 数据来源与合规范围

赛题指定和当前项目优先采用的数据源包括：

1. 上交所定期报告：<https://www.sse.com.cn/disclosure/listedinfo/regular/>
2. 深交所定期报告：<https://www.szse.cn/disclosure/listed/fixed/index.html>
3. 北交所公告：<https://www.bse.cn/disclosure/announcement.html>
4. 东方财富行业研报：<https://data.eastmoney.com/report/industry.jshtml>
5. 东方财富个股研报：<https://data.eastmoney.com/report/stock.jshtml>
6. 国家统计局宏观数据：<https://www.stats.gov.cn/sj/>

当前项目应继续遵守以下原则：

1. 真实公开数据优先
2. 所有来源可登记、可追踪
3. 补充数据必须合法合规
4. 结论要尽量回链到具体证据
5. 数据缺口要显式提示，不得静默脑补

## 7. 系统总体设计

系统总体上分三层：

### 7.1 用户产品层

负责所有用户可见的业务页面和交互体验，包括：

1. 企业问答系统
2. 数字人页面
3. 企业分析工作台
4. 企业对比判断
5. 业务总览
6. 报告与导出

设计原则：

1. 一屏看到重点
2. 页面边界清晰
3. 不堆多余入口卡
4. 结果先结论，后证据
5. 聊天页就像聊天，不像长报告

### 7.2 智能决策层

负责把用户问题转成可执行、可验证的分析流程，包括：

1. 主代理 Coordinator
2. 领域 skills
3. 检索增强
4. 记忆体系
5. 结果验证器
6. 输出组装器

设计原则：

1. 不把“多 agent 数量”当卖点
2. 默认由主代理协调，按需调用 skill
3. 输出必须附带 trace、plan、validation 等过程信息
4. 证据要可回链
5. 企业锁定失败时要自动回退，而不是报错

### 7.3 数据与工程层

负责把真实数据、模型服务、缓存、索引和治理体系串起来，包括：

1. 原始数据采集
2. 标准化清洗
3. 文本切片与索引
4. 查询与缓存
5. 在线服务
6. 任务调度
7. 数据治理
8. 审计与发布门禁

设计原则：

1. 数据链路完整
2. 来源清晰
3. 存储与索引可扩展
4. 计算与服务分层
5. 治理与质量不是可选项

## 8. 核心模块设计

### 8.1 企业问答系统

职责：

1. 围绕一个企业的一个问题做分析
2. 展示聊天消息、推理状态和证据摘要
3. 支持历史线程切换、快捷追问和线程记忆复用

当前要求：

1. 主区域必须是聊天
2. 聊天区域固定高度并内部滚动
3. 不能再出现右侧整列说明文
4. 输入后要立即出现用户消息和处理中状态，不能等接口返回才有反应
5. 必须支持常用问法、引用上轮结论、复制摘要等实用交互
6. 不能再出现顶部四块大入口卡
7. 推理过程必须紧凑，不能铺成大段文字

### 8.2 数字人页面

职责：

1. 提供独立互动场景
2. 当前以场景切换和展示位为主
3. 为后续语音、视频、WebSocket 接入预留入口

当前要求：

1. 与企业问答上下文隔离
2. 当前先不继续做聊天交互
3. 页面只保留场景、角色和扩展预留位
4. 后续需要时再单独立项深化

### 8.3 企业分析工作台

职责：

1. 呈现单企业深度分析
2. 组织经营诊断、风险和证据回链
3. 支撑正式分析与结果交付

### 8.4 企业对比判断

职责：

1. 比较多个企业的经营和风险差异
2. 输出横向结论
3. 带出证据差异和推荐动作

### 8.5 业务总览

职责：

1. 展示系统当前五条主线状态
2. 展示能力状态、数据可信度和结果可用性
3. 帮助用户理解当前系统能不能直接继续使用

注意：

`业务总览` 不能再写成项目汇报页，也不能出现“评委、老师、答辩”这类产品外文案。

## 9. 智能层详细设计

### 9.1 主代理

主代理负责：

1. 问题理解
2. 企业和对象锁定
3. 任务模式识别
4. skill 选择
5. 汇总结果
6. 触发验证
7. 回写线程

### 9.2 Skill 体系

建议继续围绕以下领域 skill 扩展：

1. 企业诊断
2. 风险预测
3. 企业对比
4. 决策建议
5. 数据治理诊断
6. 证据归因
7. 导出简报

当前已落地的重要设计：

1. `skill contract`
2. `required_payload_fields`
3. `required_evidence_keys`
4. `minimum_match_count`
5. `min_highlight_count`

这些约束用于防止“返回了一段话但没有关键字段或证据”的情况。

### 9.3 结果验证器

结果验证器已经成为主工作流的一部分，当前职责包括：

1. 检查关键字段是否返回
2. 检查企业锁定数量是否满足 skill 合同
3. 检查 highlights 是否过少
4. 检查 evidence 合同是否满足

后续应继续扩展：

1. 逻辑跳步检查
2. 风险遗漏检查
3. 来源冲突检查
4. 发布门禁前校验

### 9.4 记忆体系

当前以线程记忆为主，后续建议逐步拆分为：

1. 线程记忆
2. 公司长期记忆
3. 行业主题记忆
4. 项目记忆
5. 导出记忆

## 10. 大模型与深度学习设计

大模型在本项目中不应只承担“写答案”，还应承担：

1. 意图识别
2. Query 重写
3. 摘要生成
4. 结论归纳
5. 决策建议生成
6. 报告草稿生成

深度学习能力后续可继续往这些方向补：

1. 语义检索向量化
2. 检索重排序模型
3. 文本分类与主题识别
4. 观点 / 情绪识别
5. 风险评分模型
6. 多模态抽取

## 11. 大数据技术与大数据工程设计

### 11.1 数据链路

推荐保持以下完整链路：

1. 原始采集
2. 原始落盘
3. 文本提取
4. 标准化清洗
5. 结构化抽取
6. 切片与索引
7. 在线查询
8. 评测与治理

### 11.2 存储分层

建议分成：

1. 原始文件层
2. 结构化数据层
3. 检索索引层
4. 缓存层
5. 线程与审计层

### 11.3 工程能力体现

为了体现“大数据工程”的完整性，后续实现应重点强调：

1. 多源数据采集
2. 数据标准化
3. 异步任务与批处理
4. OLAP 分析或轻量分析引擎
5. 质量与覆盖率评测
6. 来源登记和治理
7. 审计链路

## 13. 当前主要不足

接手者应清楚，当前版本不是最终成品，还存在这些缺口：

1. 检索仍以词法 / 统计混合为主，语义检索能力还可继续加强
2. 多模态能力仍偏脚手架
3. 风险评分和更细粒度验证仍需继续补
4. 业务总览仍可继续压缩信息密度和解释成本
5. 企业分析工作台和对比页还可继续提升图表与证据表达
6. 数据工程层的异步任务、中间件和分布式扩展更多体现在设计上，尚未全部工程化落地

## 14. 交接后的优先级建议

接手者建议按下面顺序推进，不要再同时乱改前后端：

1. 稳定企业问答主链路和证据链
2. 打磨企业分析工作台和对比页
3. 继续把业务总览做成更稳定的用户向能力总览
4. 数字人模块后续如需继续，再独立做实时互动方案
5. 补强数据工程与治理链路
6. 最后再做展示包装和演示优化
