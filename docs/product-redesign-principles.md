# 2026 产品重构原则

## 1. 目标

系统不是展示技术栈，而是帮助没有金融背景的用户完成三件事：

1. 迅速看懂一家企业当前处在什么状态。
2. 能比较多家企业谁更值得跟踪。
3. 能判断结论是否可信，必要时继续追问和复核。

因此产品必须做到“十秒可上手，三分钟能产出，五分钟可追溯”。

## 2. 首页应该回答的问题

用户打开首页，第一屏只能回答下面三个问题：

1. 我现在可以分析什么对象。
2. 我应该从哪里开始提问。
3. 系统给出的判断为什么可信。

不应该把 Agent、Data、AI、架构图、比赛导向文案直接堆给用户。

## 3. 面向用户的产品主线

产品主线固定为：

1. 选择企业或企业组。
2. 提出经营、风险、对比或跟踪问题。
3. 系统返回结论、证据、建议动作。
4. 用户继续追问或导出分析材料。

所有页面都围绕这个主线组织，避免功能导航变成数据仓库式后台。

## 4. 智能体的呈现方式

智能体不是一个孤立聊天框，而是一个分析线程。

它需要同时具备：

1. 面向任务的提问入口。
2. 当前分析对象上下文。
3. 连续追问能力。
4. 过程摘要与证据引用。
5. 结果后的下一步建议。

这类设计更接近 2026 年成熟 Agent 产品的通用模式：先有资料与对象，再有问题，再生成可复用产物。

## 5. 多模态在本题中的正确位置

多模态不是单独做一个炫技模块，而是服务于“把官方披露材料变成可检索证据”。

在本题中，多模态能力的价值是：

1. 识别 PDF 年报中的表格、截图和复杂排版。
2. 将结构化抽取结果并入质量治理与证据链。
3. 弥补传统规则抽取在复杂版式上的缺口。
4. 让 Agent 的结论能引用更多正式披露字段。

因此用户界面应表达为“图表与表格已纳入证据”，而不是单独强调“我们用了多模态模型”。

## 6. 大数据工程在产品中的表达

大数据工程不是给用户展示 bronze、silver、gold 术语，而是展示四个结果：

1. 数据覆盖是否足够广。
2. 数据更新是否足够新。
3. 数据来源是否足够真。
4. 结果追溯是否足够细。

对答辩和设计报告，可以再补充底层分层、任务编排、分析仓与扩展路径。

## 7. 本项目的三层技术主线

### 7.1 数据与工程层

1. 交易所财报、东方财富研报、国家统计局宏观数据形成正式主源。
2. 统一 manifest、数据湖分层、分析仓与质量中心形成闭环。
3. 自动异常识别与人工复核保证结果可信。

### 7.2 AI 与模型层

1. 多模态文档抽取负责理解复杂 PDF。
2. 语义检索与重排负责找证据。
3. 风险预测模型负责给出量化风险先验。
4. 规则引擎负责保证基线稳定与可解释。

### 7.3 智能体与应用层

1. Agent 负责理解任务、选择工具、组织回答。
2. Web 系统负责把单点能力串成连续流程。
3. 分析材料导出负责把系统结果沉淀成正式产物。

## 8. 下一轮界面改造原则

1. 首页只保留提问入口、当前回答、来源说明和开始路径。
2. 企业分析页做成连续线程，而不是多个技术模块并排。
3. 企业对比页先回答“谁更值得跟踪”，再展开维度。
4. 数据治理页只讲覆盖、异常、复核、可信度，不讲内部实现黑话。

## 9. 资源约束下的技术策略

考虑学生设备和演示稳定性，技术路线遵循以下原则：

1. 模型以可运行的 7B 级多模态模型和轻量预测模型为主。
2. 不追求重训练超大模型，优先做高质量数据链路与可解释结果。
3. 所有高级能力必须能退化到稳定基线，保证演示不崩。
4. 创新点来自“系统闭环 + 证据链 + 多源融合 + 多模态补全”，不是单点堆名词。
