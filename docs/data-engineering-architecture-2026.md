# 数据与工程架构详细设计 2026

## 1. 工程层定位

工程层负责把真实数据、规模化处理、分布式能力、中间件和治理体系组织起来，支撑上层产品和智能能力。

目标：

1. 真实数据可信
2. 数据链路完整
3. 系统扩展性明确
4. 计算与存储架构清楚
5. 可观测、可治理、可发布

## 2. 数据来源

当前重点数据源：

1. 交易所财报
2. 东方财富研报
3. 国家统计局宏观数据
4. 行业数据与专题资料
5. 评测集与审计数据

## 3. 数据工程流程

完整流程：

1. 原始采集
2. 原始落盘
3. 清洗标准化
4. 结构化抽取
5. 切片与索引
6. 服务层缓存
7. 应用层消费

## 4. 存储架构

### 4.1 原始存储层

负责存储：

- PDF
- 原始网页
- 图片
- 报告原件

### 4.2 结构化存储层

负责存储：

- 企业信息
- 财务指标
- 报告元数据
- 审计记录
- 线程信息

### 4.3 检索索引层

负责存储：

- 文本切片索引
- 语义检索向量
- 关键词检索结构

### 4.4 缓存层

负责：

- 高频查询
- 会话状态
- 中间结果缓存

## 5. 计算架构

### 5.1 计算角色

1. 批处理任务
2. 增量更新任务
3. 在线查询任务
4. 模型推理任务
5. 分析任务

### 5.2 计算技术体现

这里体现高性能计算和大数据工程能力：

- DuckDB / OLAP 分析
- 异步任务调度
- 批量 ETL
- 索引构建
- 风险模型计算
- 多模态抽取

## 6. 分布式与中间件

### 6.1 必要中间件

1. API 服务层
2. 任务队列
3. 缓存服务
4. 会话鉴权
5. 日志系统
6. 监控告警

### 6.2 分布式体现方式

项目不需要为分布式而分布式，但要明确：

- 数据存储可扩展
- 检索索引可扩展
- 任务处理可异步扩展
- 模型服务可独立扩展

## 7. 数据治理

### 7.1 治理目标

1. 来源可信
2. 口径统一
3. 时效可见
4. 缺失可发现
5. 问题可追踪

### 7.2 治理动作

1. 覆盖率检查
2. 时效性检查
3. 缺失字段检查
4. 异常值检查
5. 人工复核队列

## 8. 发布门禁

### 8.1 目标

在系统输出进入正式展示前进行门禁控制。

### 8.2 门禁条件

1. 数据可信度达标
2. 关键证据齐全
3. 风险校验通过
4. 人工复核问题不过载
5. 交付物完整

## 9. 可观测与审计

### 9.1 可观测对象

1. 数据任务
2. 检索任务
3. 模型推理
4. agent / skill 执行链
5. 用户交互
6. 导出交付

### 9.2 审计内容

1. 谁触发了什么问题
2. 调用了哪些 skills
3. 使用了哪些证据
4. 生成了什么结论
5. 是否通过门禁

## 10. 工程层创新点

1. 数据链路全闭环
2. 证据切片与结构化索引结合
3. 检索、计算、治理一体化
4. 发布门禁机制
5. 控制面与产品面双层设计

## 11. 结论

工程层要体现的不只是“用了多少技术名词”，而是：

- 技术为什么在这里
- 它如何支撑智能层
- 它如何最终支撑用户产品层

所有工程与架构展示都必须以本文件为准。
