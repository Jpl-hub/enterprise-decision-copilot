# 架构决策

## 1. 项目主题如何避免跑偏

结论：系统必须始终围绕 `智能体赋能的企业运营分析与决策支持系统` 展开，不能退化成单一看板，也不能退化成纯聊天机器人。

当前整体架构固定为四层：

- 智能体层：意图识别、工具编排、执行轨迹、自然语言交互
- 传统应用层：FastAPI Web、业务 API、模板页面、数据库访问、权限边界
- 大数据层：多源数据采集、数据湖分层、结构化特征、批处理脚本、后续分布式扩展
- AI/深度学习层：大模型抽取与生成、语义检索、风险预测模型、多模态理解预留

这四层必须同时存在，才能把丰富度、创新点、完整度和工作量一起撑起来。

## 2. 前后端是否必须分离

结论：**不是必须做成独立部署的前后端分离**，但必须做成**逻辑分层、接口清晰、可扩展**。

当前比赛版本建议采用：

- 前端：轻量 Web 界面
- 后端：FastAPI
- 接口：JSON API
- 部署：单体部署

原因：

- 比赛更看重完整度、稳定性与演示流畅度
- 纯前后端分离会增加联调、部署和展示复杂度
- 单体部署更利于在答辩机器上稳定演示

后续如果页面复杂度明显上升，再拆成独立前端也来得及。

## 3. 数据库为什么必须上

仅靠 CSV 不够，原因：

- 后续会有多表关联
- 需要保存处理结果、缓存、标注、日志
- 需要支撑问答、筛选、检索与统计

当前阶段采用：

- SQLite + SQLAlchemy：本地稳定、零运维、便于打包
- CSV：作为导入导出和中间结果

后续可升级：

- MySQL / PostgreSQL：多人协作或线上部署时
- 向量库：做大规模研报检索时可再接入
- Spark / MinIO / Kafka：做更强的大数据演示时可升级到分布式链路

## 4. 深度学习是否要结合

结论：**要结合，但必须用在能形成评分点的地方**。

优先级从高到低建议如下：

1. 大模型做财报信息抽取、问答、报告生成
2. 预训练文本表征做研报检索与聚类
3. 风险预测模型做时间序列建模或异常检测
4. 多模态模型做年报图表、表格、文本联合理解
5. 如果数据规模足够，再尝试行业专属微调

## 5. 我们当前的技术路线

- 真实公开数据抓取
- SQLite 数据库承载结构化结果
- FastAPI Web 系统承接展示和 API
- 智能体工具链负责诊断、报告、决策简报、风险预测
- SiliconFlow 大模型负责抽取、归纳、报告生成
- 语义检索模块负责召回个股研报和行业研报证据
- 风险预测模块负责生成未来风险等级与监测指标
- 后续增加多模态理解与更强的分布式数据处理链路
