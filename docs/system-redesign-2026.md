# 2026 企业决策智能系统重新设计

## 1. 设计目标

本项目不再按“多页面 + 多模块 + 多个 agent 名称堆叠”的方式推进，而是重新定义为一个面向 2026 场景的企业决策智能系统。

核心目标不是展示“能调用多少 agent”，而是展示：

1. 传统应用如何承接稳定业务流程
2. 大模型与深度学习如何承接理解、推理、生成和多模态能力
3. 大数据技术与大数据工程如何承接真实数据、规模化处理、计算与治理
4. 三者如何组合成一个完整、专业、可验证、可交付的系统

本次重新设计追求三件事：

- 丰富度：覆盖产品、智能、数据、工程、治理、交付全链路
- 完整性：从数据采集到结论输出形成闭环
- 创新点：不是功能堆砌，而是架构和流程创新

## 2. 2026 背景下的赛题理解

2026 年的评审环境下，单纯的“聊天 + RAG + 多 agent 编排”已经很难构成强创新。真正有竞争力的方向，是把 Agent、传统应用、大模型、深度学习、大数据工程和系统治理组织成一个完整的行业系统。

这个项目应当被表述为：

“一个面向企业分析、风险判断、管理决策和交付输出的企业决策智能操作系统”

而不是：

- 一个会聊天的企业分析机器人
- 一个多 agent 的演示平台
- 一个前端堆很多卡片的大模型应用

## 3. 系统总定位

系统总定位分三层：

### 3.1 用户产品层

用户真正使用的能力只保留几条主线：

1. 企业问答系统
2. 数字人聊天室
3. 企业分析工作台
4. 企业对比判断
5. 导出与交付页面

这一层面向：

- 企业用户
- 老师与评委的业务体验
- 展示“产品价值”

这一层强调：

- 交互清晰
- 结果专业
- 证据可见
- 输出可交付

### 3.2 智能决策层

这一层是系统真正的核心。

它不再强调“有几个 agent”，而强调：

1. 领域 skills 体系
2. 任务编排与执行链
3. 检索增强与证据聚合
4. 长短期记忆
5. 决策验证
6. 风险约束
7. 结果动作化

这一层面向：

- 决策逻辑
- 领域能力
- 输出质量

### 3.3 工程与治理层

这一层负责承载系统规模、真实数据和系统可信性。

包括：

1. 数据采集与清洗
2. 多源数据存储
3. 分布式检索与索引
4. 高性能分析计算
5. 任务调度与中间件
6. 可观测性与审计
7. 数据治理与发布门禁

这一层面向：

- 系统规模
- 架构完整性
- 工程能力
- 可信与治理

## 4. 核心设计思想

### 4.1 不再把多 agent 当成主卖点

多 agent 只是组织方式，不是核心创新点。

新的原则是：

- 默认单主代理负责理解用户问题和全局协调
- 通过领域 skill 按需拉起子能力
- 子能力是“专业模块”，不是“名字不同的聊天人格”

### 4.2 从“聊天”升级到“决策闭环”

系统必须形成以下闭环：

1. 数据进入
2. 证据整理
3. 问题拆解
4. 技能执行
5. 推理与判断
6. 结论验证
7. 动作输出
8. 审计与沉淀

### 4.3 从“结果生成”升级到“结果验证”

大模型给出结论不够，必须验证：

- 是否引用真实证据
- 是否遗漏关键风险
- 是否越权脑补
- 是否与既有结论冲突
- 是否满足发布门禁

### 4.4 从“页面堆模块”升级到“界面分层”

产品界面要严格分层：

- 用户界面：问答、数字人、报告、对比、导出
- 控制界面：系统控制面、数据治理、审计、质量中心

不能再把控制面、分析面、演示面混在一页。

## 5. 产品设计

### 5.1 企业问答系统

企业问答系统是主入口，但只做一件事：围绕一个企业的一个问题给出专业回答。

目标：

- 问题聚焦
- 对话清晰
- 推理状态可见
- 证据链可追溯
- 线程历史可切换

禁止：

- 页面无限拉长
- 一屏堆太多功能块
- 把会议室或控制面硬塞进去

### 5.2 数字人聊天室

数字人聊天室是独立模块，不属于问答系统的子区域。

目标：

- 活体互动
- 多场景切换
- 轻量问答
- 陪练、彩排、娱乐扩展
- 未来接语音、视频、实时通道

### 5.3 企业分析工作台

工作台负责呈现深度分析报告，不负责承担主聊天体验。

目标：

- 单体企业完整报告
- 证据锚点
- 风险穿透
- 多模态图表与引用

### 5.4 企业对比判断

对比页负责对两个或多个企业进行横向比较。

目标：

- 指标对比
- 风险对比
- 证据差异
- 推荐结论

### 5.5 系统控制面

系统控制面不是用户主页面，而是 control plane。

面向对象：

- 老师与评委
- 项目负责人
- 系统管理员

职责：

- 展示五条主线状态
- 展示阻塞项与下一步
- 展示发布门禁
- 展示系统规模和工程能力

## 6. 智能层设计

### 6.1 主代理

主代理负责：

- 问题理解
- 任务分类
- 选择 skill
- 汇总结果
- 写入线程记忆

### 6.2 领域 skill 体系

系统应当重点建设以下 skills：

1. 财报解析 skill
2. 研报抽取 skill
3. 证据归一 skill
4. 风险穿透 skill
5. 同业比较 skill
6. 行业主题识别 skill
7. 数据质量诊断 skill
8. 管理动作生成 skill
9. 导出材料 skill
10. 决策验证 skill

### 6.3 记忆体系

记忆不应只有线程记忆。

应拆成：

1. 线程记忆
2. 公司记忆
3. 行业记忆
4. 项目记忆
5. 导出记忆

### 6.4 验证层

在输出结论前进行自动验证：

1. 证据完整性校验
2. 时效性校验
3. 风险遗漏校验
4. 结论一致性校验
5. 发布门禁校验

## 7. 大模型与深度学习设计

### 7.1 大模型职责

大模型负责：

- 问题理解
- 多步推理
- 结构化总结
- 对话交互
- 报告生成

### 7.2 深度学习职责

深度学习部分不应只写“大模型”三个字。

需要明确承担：

1. 文本语义表示
2. 检索排序
3. 文档分类
4. 情感与观点倾向识别
5. 多模态抽取
6. 风险评分模型

### 7.3 多模态能力

多模态能力用于：

- 财报图表字段抽取
- 页面截图解析
- 图像与文本联合证据
- 演示型可视化输出

## 8. 大数据技术与大数据工程设计

### 8.1 数据来源

重点围绕真实可追溯数据：

1. 交易所财报
2. 东方财富研报
3. 国家统计局宏观数据
4. 行业主题数据
5. 内部评测与审计数据

### 8.2 数据工程链路

数据工程必须是一条完整链路：

1. 数据采集
2. 原始落库
3. 清洗与标准化
4. 结构化转换
5. 证据切片
6. 索引构建
7. 缓存与服务化

### 8.3 存储架构

建议采用多层存储：

1. 原始文件存储
2. 结构化关系存储
3. 检索索引存储
4. 缓存层
5. 审计日志存储

### 8.4 计算引擎

计算层可以体现工程丰富度：

- Python 分析任务
- DuckDB / OLAP 查询
- 检索与向量索引
- 异步任务调度
- 批处理与增量更新

### 8.5 中间件

中间件不是为了写名词，而是为了系统闭环：

1. API 网关
2. 缓存层
3. 任务队列
4. 日志系统
5. 监控告警
6. 鉴权与会话系统

## 9. 创新点设计

本项目真正的创新点应当聚焦以下方面：

### 9.1 领域 skill 协议

不是通用 agent tool calling，而是企业分析领域专用 skill protocol。

### 9.2 证据图谱与回链

每个结论都能回到证据来源、时点和片段。

### 9.3 决策验证器

在回答生成后自动进行证据、逻辑和风险校验。

### 9.4 多层记忆体系

解决短对话、长项目、跨公司、跨行业的信息延续问题。

### 9.5 控制面与产品面分离

把用户产品面和系统控制面彻底拆开，体现系统架构成熟度。

### 9.6 输出动作化

输出不止结论，还包括：

- 管理动作
- 优先级
- 观察指标
- 触发条件
- 后续任务

## 10. 页面设计原则

### 10.1 企业问答页

- 单列主聊天区域为核心
- 历史可展开，不占固定右列
- 推理状态尽量薄
- 一屏先看对话

### 10.2 数字人聊天室

- 独立页面
- 强角色感
- 强场景感
- 支持未来实时化扩展

### 10.3 系统控制面

- 明确标注控制面身份
- 明确目标受众
- 显示系统主线与阻塞项
- 强调工程能力与治理能力

## 11. 评审叙事方式

项目对外讲述顺序建议是：

1. 业务价值
2. 产品结构
3. 智能决策链
4. 大数据工程体系
5. 治理与验证
6. 可交付能力

不要先讲模型名字，不要先讲用了几个 agent。

## 12. 结论

本项目后续不再按“聊天应用增强版”推进，而按“企业决策智能操作系统”推进。

后续所有设计、页面、接口和文案都需要遵守以下原则：

1. 产品面与控制面分离
2. 多 agent 退到实现层，skill 上升为核心能力
3. 大模型、深度学习、大数据工程同时体现
4. 证据、验证、治理、交付成为主线
5. 所有页面围绕“专业、完整、可信、可展示”展开
