# 2026 赛题作战手册

## 1. 官方要求基线

基于你提供的 `赛题.pdf`、`赛题相关数据下载路径文档.pdf` 以及 2025 版提交模板，2026 年这道题的刚性要求已经比较明确：

1. 作品必须是“智能体赋能的企业运营分析与决策支持系统”，智能体是核心，不是附属聊天框。
2. 必须围绕一个具体行业做深做透，且要整合多源真实数据。
3. 官方最核心的数据源是三类：交易所财报、东方财富研报、国家统计局宏观数据。
4. 必须支持自然语言问数、企业评估、风险/机会洞察、定制化报告生成、来源追溯。
5. 最终提交不仅是系统，还包括源码、部署包、设计报告、演示视频，以及额外数据集说明。
6. 决赛是答辩，评审重点至少覆盖主题创意、功能效果、技术实现、作品展示四个维度。

这意味着项目必须同时满足“能跑、能讲、能证实、能扩展”四个条件。

## 2. 高奖作品的评分抓手

从赛题文本和提交模板看，高分不在于单点炫技，而在于以下组合是否同时成立：

1. 数据真实性：所有关键分析都能追溯到原始 PDF 或公开页面。
2. 技术增量：不能只是套壳大模型，必须有自己的指标体系、Agent 工作流、数据治理和测试设计。
3. 大数据特征：要体现数据采集、数据湖/分层、批处理、可扩展计算，而不是只做小样本 notebook。
4. AI 深度：LLM、RAG、风险预测、多模态文档理解至少要形成两到三条清晰技术线。
5. 工程完整度：Web 系统、API、数据管道、部署、日志、测试、演示链路要闭环。
6. 证据化测试：准确性、稳定性、时效性、可解释性都要有量化结果或对比实验支持。

## 3. 当前仓库的真实状态

当前仓库已经具备不错的骨架：

1. FastAPI Web 系统、Agent 路由、企业诊断/报告/决策简报/风险预测链路已经成型。
2. 已接入上交所、深交所、东方财富、国家统计局的数据脚本和数据湖分层目录。
3. 已有基础评分体系、决策简报、风险预测、企业对比能力。
4. 已有测试基线，说明工程纪律是成立的。

但要冲高奖，还存在几个关键缺口：

1. 北交所官方财报链路之前没有真正补齐。
2. 数据规模仍偏小，行业覆盖还不足以支撑“高奖级”叙事。
3. 多模态文档理解、向量检索重排、分布式计算演示还没拉满。
4. 评测体系和答辩指标矩阵还需要系统化落地。

## 4. 已确认的技术路线

项目继续沿着“医药生物行业先打透，再逐步复制到更多行业”的方向推进，原因：

1. 赛题明确允许先聚焦单一行业。
2. 医药生物的财报、研报、政策、研发、风险信号都足够丰富。
3. 企业运营分析、投资研究、监管风控三类用户场景在医药行业更容易统一展示。

技术上采用四层架构：

1. 数据层：原始 PDF/HTML/CSV + bronze/silver/gold 分层 + 合规登记。
2. 分析层：指标工程、企业评分、趋势分析、风险预测、证据召回。
3. 智能体层：意图识别、工具路由、工作流编排、执行轨迹与引用证据。
4. 应用层：FastAPI + Web Dashboard + 报告接口 + 答辩展示页面。

## 5. 2026 版继续推进顺序

后续按下面顺序推进，优先级不要打乱：

1. 继续做数据基础设施：补齐北交所、扩容财报与研报样本、完善质量报告。
2. 完成财报结构化抽取升级：把 PDF 表格抽取、规则校验、异常修复体系做实。
3. 强化检索与证据链：引入切片、嵌入、重排、引用回链和答案事实核验。
4. 完善风险预测实验：LightGBM 基线 + 时序模型 + 消融实验 + 评测看板。
5. 增加大数据工程展示：Ray/Spark 风格的批处理与任务编排演示，至少形成可展示的扩展路径。
6. 最后再集中打磨文档、PPT、视频和答辩脚本。

## 7. 下一批必须落地的工程任务

我建议下一轮直接做下面三件，不再空转：

1. 建立统一的官方财报 manifest 合并脚本，把 SSE/SZSE/BSE 三所数据汇总为统一索引。
2. 给财报抽取链路补“字段级质量校验 + 异常告警 + 人工复核入口”。
3. 引入向量检索与重排，升级当前研报检索，从“关键词匹配”走向“证据级 RAG”。

这三件事做完，项目就会从“可演示骨架”进入“高奖候选系统”的阶段。

## 8. 继续参考的技术基座

后续工程演进会优先参考原始官方资料，而不是二手博客，当前重点关注：

1. AgentScope / AgentScope Studio / AgentScope Runtime：用于吸收多智能体编排、可视化追踪、运行时部署与沙箱思路。
2. OpenAI Agents SDK Tracing：用于对齐工作流追踪、Span 级调试和可观测性设计。
3. Ray Data：用于后续把大规模文档清洗、切片、批量推理、评测任务做成可扩展批处理。
4. Ray Serve：用于把检索、重排、风险预测和 Agent 工具服务化，形成更强的答辩工程叙事。

这些技术不会直接替代当前实现，而是作为下一阶段升级路线，逐步吸收到现有 Python Web 架构里。
