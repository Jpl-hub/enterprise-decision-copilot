# 特等奖级需求与系统设计总文档

更新时间：2026-03-12

## 1. 文档目标

这份文档不是展示稿，而是后续开发、重构、补数据、补模型、做可视化、写论文和准备演示时共同遵守的总设计。

它解决五件事：

1. 明确这个项目到底要做成什么，不再边做边漂。
2. 把你已经多次强调的三条主线固定下来。
3. 把产品视角、比赛视角、工程视角统一起来。
4. 把当前系统已经完成的部分和关键短板说清楚。
5. 给出接下来若干轮开发的优先级、验收标准和边界。

## 2. 项目总定位

项目名称建议统一为：

`企智策源：智能体赋能的企业运营分析与决策支持系统`

系统本质不是：

1. 一个花哨的大屏。
2. 一个只能聊天的大模型外壳。
3. 一个只会展示评分和进度条的数据后台。
4. 一个只服务比赛讲稿的演示样品。

系统本质应该是：

1. 一个面向真实用户的企业分析产品。
2. 一个以 Agent 为核心交互方式的传统应用系统。
3. 一个由高质量多源真实数据驱动的企业知识与决策中枢。
4. 一个可持续扩展到深度学习、多模态、分布式计算和数字人的完整平台。

## 3. 三条主线

### 3.1 主线一：传统应用 + Agent

这条线决定产品是否“像一个系统”，而不是像几个接口拼起来的 Demo。

必须包含：

1. 首页唯一主问答入口。
2. 企业分析页。
3. 企业对比页。
4. 数据底座与质量治理页。
5. 导出与留痕能力。
6. 线程记忆、任务模式、证据回链、执行轨迹。

最终目标：

用户提出问题后，系统不是只吐一段字，而是能识别对象、锁定任务、调用数据和模型、输出结论、给出证据、支持继续追问，并把上下文保存下来。

### 3.2 主线二：人工智能 / 深度学习 / 多模态

这条线决定系统是否真正体现 AI 和创新性。

必须包含：

1. 大模型驱动的任务理解与结果生成。
2. 财报、研报、宏观等多源证据召回。
3. 多模态财报图表解析。
4. 风险预测模型。
5. 后续可扩展的 SFT / LoRA / ModelScope 轻量微调路线。

最终目标：

让 AI 真正参与“理解、抽取、召回、判断、解释”，而不是只做一个表层聊天入口。

### 3.3 主线三：大数据技术与高质量多源大规模数据

这条线决定项目是否符合大数据赛道气质，是否有足够的工作量和工程深度。

必须包含：

1. 多源公开数据采集。
2. Bronze / Silver / Gold / Mart 分层。
3. Parquet + DuckDB 当前落地。
4. Spark-ready 批处理设计。
5. 数据质量、时效性、异常治理、人工复核。
6. 后续可继续扩样到更大企业池、更长时间跨度和更密的时序颗粒度。

最终目标：

系统不是只分析几条静态样本，而是具备持续扩样、持续计算、持续治理的工业化数据骨架。

## 4. 核心设计原则

### 4.1 用户可见界面必须是产品语言

用户页面里不应该出现：

1. `比赛`
2. `答辩`
3. `赛题`
4. 任何面向评委而不是面向用户的字眼

这些词可以出现在内部文档、项目管理和比赛材料中，但不能污染产品界面。

### 4.2 每个页面只能有一个主任务

1. 首页：问问题、切企业、跳转。
2. 企业分析页：看结果、看风险、看证据。
3. 企业对比页：横向比较、证据验证。
4. 数据底座页：看系统、看数据、看治理。
5. 导出页：生成分析材料。

禁止一个页面出现两套相同职责的交互。

### 4.3 文案必须为决策服务

1. 一张卡片最多一句说明。
2. 没有行动价值的小灰字一律删除。
3. 标题要清楚，不要堆术语。
4. 所有说明都要能回答“用户看完之后下一步做什么”。

### 4.4 可视化必须可点击、可解释、可回链

图表不是装饰。

图表至少满足其一：

1. 能跳转到企业页。
2. 能触发追问。
3. 能打开证据。
4. 能打开治理详情。

### 4.5 所有结论必须有证据

证据至少来自下列之一：

1. 官方财报。
2. 个股研报。
3. 行业研报。
4. 宏观指标。
5. 多模态财报图表锚点。
6. 风险模型产物。

## 5. 目标用户与核心场景

### 5.1 企业管理者

核心需求：

1. 快速看懂企业经营状态。
2. 找到风险和优先动作。
3. 结合行业和宏观看经营环境。
4. 导出结构化分析材料。

### 5.2 投研分析人员

核心需求：

1. 对单家企业做经营分析。
2. 对两家或多家企业做横向对比。
3. 看证据是否可靠、是否新鲜。
4. 沿着问题持续追问。

### 5.3 风控与数据治理角色

核心需求：

1. 识别数据覆盖缺口和质量异常。
2. 识别高风险企业和风险驱动项。
3. 检查模型和结论是否可信。
4. 跟踪人工复核与治理进度。

## 6. 产品形态总览

### 6.1 首页：企业运营分析中枢

首页是唯一主问答入口。

承载的内容应该是：

1. 企业切换。
2. 主问答输入。
3. 4 个以内的核心跳转卡。
4. 一组高价值可视化。
5. 少量高价值情报卡。

不应该承载：

1. 第二套问答。
2. 大量说明性文字。
3. 平铺式堆卡片。
4. 面向评审的文案。

### 6.2 企业分析页：结果与证据页

必须显示：

1. 企业核心结论。
2. 经营分析。
3. 风险画像。
4. 证据资料。
5. 多模态锚点。
6. 导出入口。

必须删除：

1. 第二套问答区。
2. 重复的任务流。
3. 与首页重复的入口。

### 6.3 企业对比页：横向判断台

必须显示：

1. 双企业或多企业对比结论。
2. 经营分数与风险矩阵。
3. 财报时效差异。
4. 研报热度与观点差异。
5. 多模态证据锚点。

### 6.4 数据底座页：系统控制塔

必须显示：

1. 系统三主线完成度。
2. 湖仓分层与数据规模。
3. 定期披露覆盖。
4. 质量热点字段。
5. 多模态抽取进度。
6. 训练数据与模型产物。
7. 计算作业与 Spark-ready 路线。

### 6.5 导出页：分析材料中心

必须显示：

1. 结构化章节提纲。
2. 引用清单。
3. 证据包路径。
4. 可继续转为 DOCX / PPT 的中间稿。

## 7. 功能需求分层

### 7.1 P0 必须稳定成立

1. 用户能从首页持续提问。
2. 用户能进入企业分析页看到完整结果。
3. 用户能进入企业对比页。
4. 用户能看到数据时效。
5. 所有结论能回到至少一种公开来源。
6. 线程与任务模式不会丢。

### 7.2 P1 必须形成亮点

1. 多模态财报图表锚点进入主分析链。
2. 风险模型进入企业分析与对比链。
3. 数据底座页可展示真实规模、治理和路线。
4. 核心图表可点击跳转。
5. 导出材料可直接作为正式汇报稿基础。

### 7.3 P2 冲击特等奖的关键项

1. 企业池扩样。
2. 年报以外的季度持续更新。
3. SFT 数据集扩充与轻量微调试验。
4. Spark 或同级批处理接口接入。
5. 数字人 / Unity 作为后续增强层而不是主链依赖。

## 8. 数据设计

### 8.1 数据源策略

系统必须坚持真实、公开、可追溯。

当前主数据源：

1. 交易所定期报告。
2. 东方财富个股研报。
3. 东方财富行业研报。
4. 国家统计局宏观指标。

后续扩展数据源建议：

1. 官方公告与互动易类公开文本。
2. 国家药监、医保、招采等行业政策公开信息。
3. 行业协会、政府统计和重点专题报告。
4. 企业官网公开新闻与投资者关系材料。

### 8.2 数据质量要求

每一类数据都必须有：

1. 来源登记。
2. 时间戳。
3. 去重策略。
4. 质量报告。
5. 缺失和异常标记。
6. 人工复核入口。

### 8.3 数据规模目标

当前真实状态，基于 2026-03-12 本地仓：

1. 仓内表和视图：17。
2. 湖仓总行数：9649。
3. 财务特征：55 行。
4. 个股研报：309 行。
5. 行业研报：1308 行。
6. 宏观指标：11 行。
7. 官方定期报告清单：21 行。
8. 扩展企业池报告：2783 行。

这说明当前系统已经有“可运行的数据底盘”，但离“高质量多源大规模数据”还有明显差距。

目标分三档：

1. 比赛可讲解档：当前 6 家核心企业 + 数百到数千条外围证据。
2. 高奖工程档：扩到 50 至 200 家企业，覆盖 3 到 5 年时序。
3. 顶级展示档：扩到 500 家以上企业池，形成跨行业、跨阶段、跨任务的大样本仓层。

### 8.4 时效性要求

系统不能只停在年度报告。

必须持续覆盖：

1. 年报。
2. 一季报。
3. 半年报。
4. 三季报。
5. 最新个股研报。
6. 最新行业研报。
7. 最新宏观窗口。

当前真实状态：

1. 年报覆盖：6/6，最新披露日 2025-04-29。
2. 一季报覆盖：6/6，最新披露日 2025-04-29。
3. 半年报覆盖：6/6，最新披露日 2025-08-30。
4. 三季报覆盖：6/6，最新披露日 2025-10-30。

## 9. 大数据与计算架构

### 9.1 当前落地架构

当前已经形成：

1. `raw`
2. `bronze`
3. `silver`
4. `gold`
5. `mart`
6. `DuckDB`
7. `Parquet`

这是一个单机可运行湖仓。

当前优点：

1. 本机可稳定演示。
2. 计算链路清楚。
3. 数据与结果可回溯。
4. 质量报告可直接生成。

当前短板：

1. 还不是大规模分布式计算。
2. 增量调度能力还弱。
3. 企业池规模仍偏小。
4. 时序特征与大样本训练集还不足。

### 9.2 下一阶段大数据路线

建议按三步走：

#### 第一步：把单机湖仓做厚

1. 扩企业池。
2. 扩年份。
3. 扩公告和行业政策。
4. 扩季度频率。
5. 扩治理规则。

#### 第二步：把批处理做强

1. 保留当前 `python + duckdb`。
2. 所有关键作业补齐 Spark-ready 清单。
3. 统一分区键和作业输入输出协议。
4. 引入增量更新和失败重试设计。

#### 第三步：把分布式接口做实

推荐路线：

1. 数据湖：Parquet / MinIO 或对象存储。
2. 计算：Spark。
3. 调度：Airflow / Prefect。
4. 查询：DuckDB + ClickHouse 或同级 OLAP。
5. 检索：全文索引 + 向量索引。

### 9.3 当前已设计的 Spark-ready 作业

当前本地作业清单已有 5 个：

1. `bronze_to_silver_governance`
2. `silver_to_gold_company_fact`
3. `multimodal_extract_batch`
4. `multimodal_sft_dataset_build`
5. `risk_model_training`

这 5 个作业就是后续大数据赛道的主干。

## 10. Agent 总体架构

### 10.1 Agent 角色定义

系统不建议一开始就上复杂多智能体群聊，而是采用“主编排 Agent + 专项分析器”结构。

主 Agent 负责：

1. 理解问题。
2. 识别任务模式。
3. 锁定企业对象。
4. 组装证据。
5. 调用专项服务。
6. 汇总输出。

专项分析器负责：

1. 企业经营诊断。
2. 风险预测。
3. 决策建议。
4. 行业趋势。
5. 数据治理与可信度说明。
6. 多模态财报证据补全。

### 10.2 Agent 状态机

每次任务应遵循统一状态机：

1. `问题进入`
2. `对象锁定`
3. `任务识别`
4. `证据召回`
5. `模型计算`
6. `结果生成`
7. `证据回链`
8. `线程存档`

### 10.3 线程与记忆

线程记忆至少保留：

1. 当前企业。
2. 当前任务模式。
3. 历史问题。
4. 最近证据。
5. 最近结论。
6. 用户是否在做对比、治理还是导出。

### 10.4 Agent 输出标准

每轮输出都应尽量包含：

1. 一句结论。
2. 2 到 4 条关键判断。
3. 2 到 4 条动作建议。
4. 证据摘要。
5. 可信度说明。
6. 推荐追问。

## 11. AI / 深度学习设计

### 11.1 大模型层

定位：

1. 问题理解。
2. 任务分解。
3. 证据整合。
4. 分析生成。
5. 结果压缩。

不建议让大模型直接做的事情：

1. 替代全部结构化计算。
2. 直接编造财务结论。
3. 不带证据地输出高风险判断。

### 11.2 多模态层

定位：

1. 解析复杂财报页面。
2. 补齐纯文本抽取无法稳定获取的字段。
3. 输出页锚点和页面图像链接。

当前真实状态：

1. 多模态抽取结果：8/21。
2. SFT 数据集：8 条。
3. 已进入企业分析和企业对比证据流。

当前缺口：

1. 覆盖率仍偏低。
2. 字段稳定性还要提升。
3. 训练数据量远不够微调规模。

### 11.3 风险模型层

定位：

1. 风险等级预测。
2. 高风险概率预测。
3. 关键因子解释。

当前真实状态：

1. 已有表格风险模型产物。
2. 当前模型文件包括 `risk_tabular_model.json`、`risk_tabular.joblib`、`risk_tabular_metadata.json`。
3. 序列模型产物尚未稳定落盘。

### 11.4 微调与 ModelScope 路线

本项目的微调路线必须克制，不做无意义的大规模训练口号。

建议顺序：

1. 先做高质量 SFT 数据集。
2. 再做轻量 LoRA / QLoRA。
3. 优先试验财报多模态抽取和风险解释两个方向。
4. 使用 ModelScope / ms-swift 做轻量微调实验和模型管理。

下面这些资料用于指导架构，而不是照抄界面。

   借鉴点：Agent、handoffs、sessions、tools、guardrails 的正式编排方式。

   借鉴点：持久化工作流、状态图、可恢复执行。

   借鉴点：多 Agent 协作、任务状态和长链路编排。

   借鉴点：模型托管、多模态模型接入、训练与推理生态。

1. [Apache Spark 官方文档](https://spark.apache.org/docs/latest/)
   借鉴点：大规模批处理、Structured Streaming、可扩展计算框架。

## 15. 数字人 / Unity 的位置

这个方向可以做，但绝对不能提前喧宾夺主。

正确定位是：

1. 它是展示增强层。
2. 它可以承接语音播报、讲解和交互引导。
3. 它不能代替核心分析能力。
4. 它必须建立在数据、Agent、模型、证据链都稳定之后。

推荐落位：

1. 第二阶段加入语音讲解人设。
2. 第三阶段接 Unity 数字驾驶舱。
3. 第四阶段接入 3D 企业运营沙盘。

## 16. 分阶段路线图

### 阶段 A：系统收敛

目标：

1. 收干净首页和企业页。
2. 去除面向评审的界面词。
3. 固定信息架构和设计规范。

验收：

1. 首页只有一个问答入口。
2. 企业页没有第二问答区。
3. 页面标题、按钮和图表结构稳定。

### 阶段 B：数据和证据补强

目标：

1. 继续扩充季度和行业数据。
2. 把多模态从 8/21 往全量推进。
3. 扩展企业池和外围证据。

验收：

1. 核心样本全覆盖。
2. 多模态覆盖率明显提升。
3. 企业对比和企业分析都能稳定引用新证据。

### 阶段 C：模型和计算补强

目标：

1. 做 SFT 数据集扩充。
2. 做 ModelScope / ms-swift 轻量实验。
3. 做风险模型产物管理。
4. 做 Spark-ready 批处理接口实装。

验收：

1. 有真实训练数据规模。
2. 有可复用模型产物。
3. 有可展示的批处理链路。

### 阶段 D：展示增强

目标：

1. 做更强的驾驶舱图层。
2. 做更强的轨迹与证据回放。
3. 视情况接语音和数字人。

验收：

1. 可视化不再是摆设。
2. 每个图层都能解释业务。
3. 增强层不破坏产品简洁性。

## 17. 验收指标

### 17.1 产品指标

1. 首页 5 秒内让用户明白可以做什么。
2. 任一核心页面不出现重复主动作。
3. 关键按钮对齐，主次明确。
4. 页面内无无意义灰字堆砌。

### 17.2 数据指标

1. 核心企业财报覆盖率接近 100%。
2. 季度披露覆盖率接近 100%。
3. 研报与行业数据保持更新。
4. 关键热点空值率逐步下降。

### 17.3 AI 指标

1. 大模型输出始终带证据。
2. 多模态抽取覆盖率持续增长。
3. 风险模型有稳定产物和可解释输出。
4. SFT 数据规模持续扩张。

### 17.4 大数据指标

1. 湖仓作业可重复运行。
2. 分区、作业和产物结构统一。
3. Spark-ready 作业定义完整。
4. 扩样不会推翻已有架构。

## 18. 结论

后续开发必须遵守这份文档的根逻辑：

1. 先把系统做成真正服务用户的产品。
2. 再把 Agent、深度学习、多模态和大数据能力做深。
3. 比赛视角是约束和目标，但不能让产品界面和系统结构为比赛口号服务。
4. 所有加分项都要能回到真实数据、真实能力、真实工作量。

一句话概括：

`先做成一个优秀的企业智能分析系统，再把它做成一个足以拿特等奖的作品。`
