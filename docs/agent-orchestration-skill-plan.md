# 2026 Agent 编排与 Skill 建设计划

## 1. 这个项目离 2026 大厂 Agent 规范还有多远

当前项目已经具备一套可运行的 Agent 骨架：

1. 有任务路由、工具调用、线程持久化和失败兜底。
2. 有企业焦点对象、连续追问和分析轨迹。
3. 有风险预测、决策建议、数据质量等明确分析能力。

但如果要拿去面试，不应只讲“我做了一个聊天分析系统”，而要讲成一套标准 Agent 工程：

1. 有线程级状态管理，而不是临时拼 prompt。
2. 有 skill 边界，而不是所有问题都落到一个大工具里。
3. 有上下文承接规则，而不是上一轮状态在前端乱传。
4. 有 trace、plan、fallback、评测、回放和观测闭环。
5. 有前端工作台和后端编排的一致职责划分。

## 2. 本项目的 Agent 编排定位

本项目不做开放式通用聊天助手，而是做领域型分析 Agent。

因此编排目标固定为：

1. 锁定企业或企业组。
2. 判断任务类型。
3. 选择对应 skill 或工具链。
4. 产出结论、证据、动作建议。
5. 保留线程状态，承接下一轮追问。

这比“单次问答”更接近企业内部分析 Copilot 和 Notebook 式工作流。

## 3. skill 的正确颗粒度

当前项目适合拆成下面几类 skill，而不是做泛化插件市场：

1. `focus_company_skill`
   负责对象锁定、企业别名消歧、线程焦点保持。
2. `diagnosis_skill`
   负责经营状态拆解、问题诊断、核心变量判断。
3. `risk_forecast_skill`
   负责三层风险拆解、风险模型调用、监测项生成。
4. `decision_brief_skill`
   负责动作建议、经营判断、管理层视角输出。
5. `evidence_trace_skill`
   负责证据回链、来源引用、研报与财报证据拼装。
6. `quality_guard_skill`
   负责数据覆盖、异常分布、人工复核入口提示。

面试时要强调：`skill 不是 UI 按钮，而是后端可组合能力单元`。

## 4. 上下文管理必须加强的四层

### 4.1 线程上下文

线程至少要稳定保存：

1. 当前聚焦企业。
2. 上一轮任务模式。
3. 最近一次有效结论。
4. 最近一次失败状态。

### 4.2 追问改写

连续追问不要完全依赖前端传 `task_mode`。

后端要负责：

1. 将“这家公司”“它”替换成当前焦点企业。
2. 对“展开讲讲”“具体一点”“继续”这类模糊追问承接上一轮任务模式。
3. 当本轮出现明确新意图时，允许切换 skill。

### 4.3 证据上下文

后续还要补：

1. 最近引用过的财报字段。
2. 最近引用过的研报证据。
3. 最近输出过的风险监测项。

这能减少重复检索，也让追问更像连续分析而不是重新开题。

### 4.4 UI 上下文

前端只负责显示线程和允许手动选模式，不应该偷偷把上一轮模式强制塞回后端。

否则会出现：

1. 用户明明想切新问题，系统却继续沿用旧链路。
2. 页面状态和后端状态不一致。
3. 追问逻辑看起来像坏了。

## 5. 2026 参考规范

当前主流 Agent 工程的共同点不是谁家 API 名字更响，而是下面这些规范趋同：

1. 线程或 session 持久化。
2. tool / skill 显式编排。
3. trace / run 级观测。
4. memory 与 context window 分层管理。
5. 失败可恢复而不是直接报错。
6. 有评测与回放，不靠主观感觉说“效果不错”。

本项目要对齐这套规范，而不是只追求页面像聊天产品。

## 6. 下一轮工程动作

1. 把现有 tool 显式提升为 skill registry，形成可扩展编排表。
2. 增加线程级 `last_task_mode`、`last_task_label` 与追问承接规则。
3. 增加关键问题集评测，覆盖诊断、风险、决策、追问切换四类场景。
4. 给 trace 增加可回放视角，便于比赛答辩和面试展示。
5. 给证据链增加引用折叠区，让前端只展示需要的结果，不堆实现细节。
