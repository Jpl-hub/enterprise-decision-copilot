# 技术栈、创新点与数据库设计

## 1. 当前技术栈

### 1.1 应用层

1. FastAPI: 提供企业分析、风险预测、数据治理、Agent、材料导出、审计等正式接口。
2. Vue 3 + Pinia + Vue Router: 构建前端产品界面、状态管理与页面导航。
3. Vite: 前端构建与开发环境。

### 1.2 智能体层

1. 自研 Agent 工作流: 负责意图识别、任务拆解、工具选择、线程管理、执行轨迹与计划输出。
2. 线程化上下文: 支持同一企业下的连续追问与历史保留。
3. 持久化分析线程: Agent 会话落库，服务重启后仍可延续同一条分析线程。
4. 证据化输出: Agent 回答与财报、研报、风险模型和质量治理结果联动。

### 1.3 AI 与模型层

1. 大语言模型接口: 用于生成分析摘要、决策简报、结构化问答。
2. 多模态视觉模型接口: 用于官方年报 PDF 中表格、图表、复杂版面抽取。
3. 风险预测模型: 当前以轻量可运行的表格模型为主，支持风险概率与关键驱动输出。
4. 语义检索与证据召回: 用于把研报和资料摘要接入决策链路。

### 1.4 数据工程层

1. SQLite: 承担轻量应用库、用户与会话、线程与审计、基础结构化数据承载。
2. DuckDB: 承担分析仓与多表查询，适合本项目的离线分析与展示。
3. CSV/JSON/PDF: 承担公开原始数据、清洗结果、抽取结果和质量记录。
4. 批处理脚本: 负责财报下载、研报抓取、抽取构建、质量扫描与训练数据准备。

## 2. 目前明确的创新点

### 2.1 不是“聊天框”，而是企业分析 Agent

系统不是把大模型嵌成一个聊天框，而是让 Agent 进入完整企业分析流程：

1. 识别企业对象。
2. 拆解当前任务。
3. 调用报告、风险、治理、对比等工具。
4. 输出结论、证据和后续动作。
5. 保留线程，支持继续追问。

这是产品级 Agent，而不是 API 套壳。

### 2.2 多模态不是炫技模块，而是证据补全能力

系统没有把“多模态”单独做成噱头，而是把它嵌入正式证据链：

1. 识别官方 PDF 年报里的复杂表格和图表。
2. 将抽取结果纳入质量治理。
3. 把补全字段用于 Agent 回答和分析材料导出。
4. 对低覆盖结果自动生成复核工单。

这让多模态真正服务于决策可信度。

### 2.3 大数据工程不是后台名词，而是可感知的可信度系统

我们没有只展示 bronze/silver/gold 术语，而是把数据工程结果做成用户能判断的四个维度：

1. 覆盖率。
2. 来源真实性。
3. 异常与缺失情况。
4. 是否进入复核闭环。

这让大数据工程能力直接体现在产品体验中。

### 2.4 规则、模型、Agent 三层协同

系统不是只靠一个模型出答案，而是三层协同：

1. 规则层: 保证底线稳定与可解释。
2. 模型层: 提供风险概率、多模态抽取和语义理解能力。
3. Agent 层: 负责把这些能力组织成任务流程。

这种结构更适合比赛，也更像真实企业系统。

### 2.5 线程持久化与操作审计进入正式主链路

系统已经不再依赖纯内存会话，而是把关键过程落进应用库：

1. Agent 线程可持续保留。
2. 用户关键操作可审计。
3. 导出、复核、问答都能追踪到用户与对象。

这让系统更接近真实企业应用，而不是一次性演示品。

### 2.6 角色权限控制采用标准 RBAC 思路

权限设计不搞自创体系，而是采用业界常见的角色分层：

1. admin: 管理员，可查看审计、自动同步复核、导出、治理。
2. analyst: 分析员，可导出、提交复核、开展分析。
3. viewer: 只读用户，可看分析与治理结果，但不能执行写操作。

这种方式更清晰，也方便后续扩展团队协作。

## 3. 数据库设计说明

## 3.1 为什么不是只用一种数据库

本项目采用“应用库 + 分析仓”分工，而不是把所有东西硬塞进一个数据库：

1. SQLite 负责轻量事务型数据: 用户、会话、线程、审计、基础结构化表。
2. DuckDB 负责分析型数据: 多表汇总、统计、离线分析视图。

这样设计的原因是：

1. 学生资源下足够轻量。
2. 工程上清楚，便于部署。
3. 既能支撑正式产品，又能支撑数据分析。
4. 后续可以平滑迁移到 PostgreSQL + 湖仓体系。

## 3.2 SQLite 当前核心表

### users

用途: 用户身份与角色管理

字段:
1. user_id: 用户主键
2. username: 登录用户名，唯一
3. display_name: 展示名称
4. password_hash: 密码哈希
5. password_salt: 密码盐值
6. role: 用户角色，例如 admin、analyst、viewer
7. created_at: 创建时间
8. last_login_at: 最近登录时间

### user_sessions

用途: Bearer Token 会话管理

字段:
1. token: 会话 token 主键
2. user_id: 关联用户
3. created_at: 创建时间
4. expires_at: 失效时间
5. revoked_at: 注销时间

### agent_threads

用途: 持久化 Agent 分析线程

字段:
1. thread_id: 线程主键
2. user_id: 线程所属用户
3. title: 线程标题
4. focus_company_code: 当前关注企业代码
5. focus_company_name: 当前关注企业名称
6. created_at: 创建时间
7. updated_at: 最近更新时间

### agent_messages

用途: 线程消息历史

字段:
1. message_id: 消息主键
2. thread_id: 所属线程
3. role: user 或 assistant
4. content: 消息内容
5. created_at: 创建时间

### audit_logs

用途: 关键操作审计

字段:
1. log_id: 审计主键
2. user_id: 操作用户
3. event_type: 操作类型
4. target_type: 目标类型
5. target_id: 目标对象
6. detail_json: 细节 JSON
7. created_at: 操作时间

### companies

用途: 企业主数据表

字段:
1. company_code
2. company_name
3. exchange
4. industry
5. segment

### financial_features

用途: 年报核心财务特征表

字段涵盖:
1. 收入、利润
2. 毛利率、净利率、ROE
3. 研发强度
4. 债务与流动性
5. 周转与现金流
6. 来源链接与披露时间

### research_reports / industry_reports / macro_indicators

用途:
1. 承接公开研报与行业观点
2. 承接宏观指标
3. 为 Agent 和证据链提供补充资料

## 3.3 DuckDB 的角色

DuckDB 不是替代 SQLite，而是承担分析仓功能：

1. 汇总财务特征、研报热度、行业热度。
2. 提供高性能的只读分析视图。
3. 支撑企业对比、全局看板、离线统计和答辩材料准备。

## 4. 为什么要补登录注册

这个系统现在已经不是静态可视化页，而是包含：

1. 企业分析结论。
2. 风险判断。
3. 多源数据治理。
4. 材料导出。
5. Agent 会话线程。

如果没有登录注册，产品就是裸露业务接口，既不专业，也不符合企业系统常识。因此已补充：

1. 注册。
2. 登录。
3. token 会话。
4. 受保护 API。
5. 前端路由守卫与退出登录。

## 5. 后续可以继续增强的方向

1. 团队空间: 不同小组维护不同企业观察清单与分析结论。
2. 线程检索与归档: 支持按企业、时间、问题类型检索历史分析线程。
3. 权限后台: 可由管理员调整角色，而不是只靠注册默认分配。
4. 审计筛选: 按用户、事件类型、时间范围检索操作日志。
